01

What async UI taught us

When Promises and async/await went mainstream, the entire mental model of building UI had to shift. We moved from synchronous "request → wait → render" to managing a new class of problems: loading states, error boundaries, race conditions, stale data, partial failures.

That transition took the industry years to get right. We built skeleton loaders, React Suspense, React Query, optimistic updates — all around one core assumption:

The response arrives whole, or not at all.

Async UI had four states. Clean. Predictable. A state machine you could draw in 30 seconds:

Async state machine
idle → loading → complete | error
Streaming state machine
idle stream-started stream-in-progress stream-interrupted stream-errored-partial stream-errored-empty stream-complete

Every one of those streaming states needs a deliberate design decision. Most teams today only model three of them.

02

How streaming UI actually works

At the protocol level, streaming runs via Server-Sent Events (SSE) — the server holds the HTTP connection open and pushes token deltas as they're generated. Each event looks like:

SSE stream 
// server pushes these continuously
data: {"delta": {"text": "The"}}
data: {"delta": {"text": " abstraction"}}
data: {"delta": {"text": " layer"}}
data: [DONE]

On the client you consume this via the ReadableStream API on the fetch response body:

React — naive approach 
const response = await fetch('/api/chat', { method: 'POST' });
const reader = response.body.getReader();
const decoder = new TextDecoder();

while (true) {
  const { done, value } = await reader.read();
  if (done) break;

  const token = parseChunk(decoder.decode(value));
  // setState 20-30x per second — this causes problems
  setStreamedText(prev => prev + token);
}

The better pattern — accumulate in a ref, render via requestAnimationFrame:

React — production approach 
const bufferRef = useRef('');
const [displayText, setDisplayText] = useState('');

// accumulate synchronously — no React batching issues
bufferRef.current += token;

// sync to state at a controlled frame rate
requestAnimationFrame(() => {
  setDisplayText(bufferRef.current);
});
03

The gotchas that hit you in production

These aren't tutorial problems. These are the things that show up after you ship.

GOTCHA 01
Markdown flickering

Tokens render mid-syntax. Users watch **The best snap to bold when the closing ** arrives. Code blocks appear as raw text then suddenly reformat. The fix: buffer to semantic boundaries before rendering. Claude's UI does this — that's why it feels coherent mid-stream.

GOTCHA 02
Component unmounting mid-stream

User navigates away while a stream is in flight. You're calling setState on an unmounted component — silent memory leak in React 18. Always wire an AbortController to the effect cleanup function. Almost every first implementation forgets this.

GOTCHA 03
Errors arriving mid-stream

80% of a response rendered, then a rate limit or timeout hits. Most apps wipe the partial content and show an error state. That's wrong — the user already read half of it. Preserve what arrived, append an interruption indicator.

GOTCHA 04
Scroll anchoring vs user intent

Auto-scrolling to the bottom while the user tries to scroll up. You need to detect manual scroll intent and stop fighting them. Track whether scrollHeight - scrollTop === clientHeight and only auto-scroll when they're already at the bottom.

GOTCHA 05
Race conditions on re-submission

User sends a message, edits mid-stream, sends again. Two streams updating the same state — whichever resolves last wins. Same pattern as the classic async race condition: use a request sequence number and discard tokens from stale requests.

GOTCHA 06
Accessibility is broken by default

aria-live="polite" on a region updating 30× per second either overwhelms or ignores screen readers entirely. Announce a status separately and only surface the completed content. This is the "add keyboard support later" of streaming UI — it almost never makes v1.

04

What major products are actually doing

Open DevTools on any of these, go to Network → EventStream, and watch the raw SSE deltas arrive. Then look at the UI and reverse-engineer every design decision made between those two layers. It's one of the most educational exercises available right now.

Claude Anthropic

Buffers to semantic boundaries — complete sentences, closed code blocks — before rendering. Trades raw speed perception for visual coherence. The most considered streaming UX of the major products.

ChatGPT OpenAI

Streams aggressively — tokens appear almost immediately. Tradeoff: visible markdown flicker in earlier versions, improved now. Stop-generation preserves partial content well.

Vercel v0 Vercel

Streams code, not prose. Shows a loading skeleton while code streams in the background, reveals only on a valid parseable boundary. Hides the stream entirely for correctness.

GitHub Copilot GitHub

Ghost text pattern — streamed content appears greyed-out, visually separated from committed content. The cleanest UX separation of "in-flight" vs "accepted" content in any streaming product today.

Perplexity Perplexity

Two parallel streams: answer + citations, kept correlated so citation markers appear near the claims they support. Genuinely hard engineering — different arrival rates, must stay semantically linked.

The engineers who get this right

…aren't the ones who know the most about LLMs. They're the ones who already got async UI right — and had the discipline to ask "what assumptions am I carrying over that no longer apply?"

The answer, it turns out, is most of them. Model your states explicitly, upfront, before you write a single line of streaming code. Everything else follows.
